A3Nodes is the internal pipeline architecture behind Yosoi's native support for JavaScript-rendered pages.

## Overview

Standard Yosoi fetches raw HTML over HTTP. A3Nodes adds a second fetcher path that drives a real browser (Playwright), waits for the DOM to settle, then passes the rendered HTML into the same discovery and extraction pipeline.

L2 sites (SPAs, data-driven apps, framework-rendered pages) work without any changes to your contract or scrape code.

```python
import yosoi as ys

class Product(ys.Contract):
    title: str = ys.Title()
    price: str = ys.Price()

# DOM fetcher can be selected automatically for JS-heavy sites,
# or forced explicitly (API subject to change in beta)
result = await ys.scrape("https://example-spa.com/product/42", contract=Product)
```

## Pipeline design

The A3Nodes pipeline inserts a DOM node before discovery:

```
URL
 └─► FetchNode (HTTP or Browser)
       └─► CleanNode (HTML noise reduction)
             └─► DiscoveryNode (LLM selector discovery)
                   └─► ExtractionNode (BeautifulSoup)
                         └─► ValidationNode (Pydantic)
```

For static HTML, `FetchNode` uses the lightweight HTTP fetcher. For JS-rendered pages, it launches a browser context, navigates to the URL, waits for network idle, and captures the DOM. Everything downstream is the same.

Smart detection is also planned: Yosoi will inspect the raw HTTP response and automatically escalate to the browser fetcher if the page looks like a JS shell.

## Current status

The DOM fetcher is in **active beta**. Core functionality works but the public API may change before it hits stable.

Known limitations:
- No support for authenticated sessions or multi-step navigation yet
- Browser context is created per-scrape (no persistent sessions)
- Smart auto-detection is not yet enabled; you have to opt in explicitly

Follow progress on the [Roadmap](/roadmap/) and the [GitHub project](https://github.com/orgs/CascadingLabs/projects).

## Why "A3Nodes"?

The name comes from the **async agent architecture** used internally. Each pipeline stage is a discrete async node that can be swapped or replaced on its own. The DOM fetcher is one node. Future nodes (authenticated fetchers, proxy rotation, custom wait strategies) plug into the same chain.
